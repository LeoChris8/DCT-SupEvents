# §6.4 — Vue des données

## Modèle entité-relation

SupEvents repose sur huit entités principales qui structurent l'ensemble du domaine métier de la plateforme. Le modèle distingue clairement les acteurs (User, Organizer), les objets métier (Event, Ticket, Payment) et les entités de support (Notification, Category, WaitingListEntry). Toutes les relations passent par des entités de plein droit — notamment Ticket, qui matérialise l'inscription d'un utilisateur à un événement et porte la logique de paiement associée.

Le modèle place **Ticket** et **Payment** comme entités centrales de la transaction : un Ticket représente l'intention d'inscription, et un Payment matérialise le règlement associé. L'entité **Organizer** est distincte de **User** pour permettre la validation administrative (champ `status`, colonne `validated_by`). La relation **User ↔ Event** est volontairement indirecte — elle transite toujours par Ticket — ce qui garantit la traçabilité complète des inscriptions. L'entité **WaitingListEntry** gère nativement les listes d'attente sans polluer la table Ticket de statuts ambigus.


## Dictionnaire de données

### Entité `User`

| Champ | Type | Contraintes | Description | Sensibilité RGPD |
|---|---|---|---|---|
| id | UUID | PK, NOT NULL | Identifiant unique de l'utilisateur | Non |
| email | VARCHAR(255) | UK, NOT NULL | Adresse email de connexion | **Oui** |
| password_hash | VARCHAR(255) | NOT NULL | Hash bcrypt du mot de passe | **Oui** |
| first_name | VARCHAR(100) | NOT NULL | Prénom | **Oui** |
| last_name | VARCHAR(100) | NOT NULL | Nom de famille | **Oui** |
| phone | VARCHAR(20) | NULL | Numéro de téléphone | **Oui** |
| is_active | BOOLEAN | NOT NULL, DEFAULT true | Compte actif ou désactivé | Non |
| created_at | TIMESTAMP | NOT NULL, DEFAULT now() | Date de création du compte | Non |
| updated_at | TIMESTAMP | NOT NULL, DEFAULT now() | Dernière mise à jour | Non |

> **Note RGPD** : `email`, `first_name`, `last_name`, `phone` sont des données personnelles directement identifiantes. Stratégie : chiffrement applicatif AES-256 pour `phone`, pseudonymisation lors de l'export analytics, durée de rétention 3 ans après dernière connexion puis purge ou anonymisation. `password_hash` : bcrypt cost ≥ 12, jamais exposé en API.

## Stratégie de stockage

**PostgreSQL** héberge toutes les entités transactionnelles : User, Organizer, Event, Ticket, Payment, Category, WaitingListEntry et RefreshToken. Le choix du relationnel est justifié par la nécessité de transactions ACID sur les opérations critiques — en particulier l'atomicité du couple décrément de `tickets_sold` + création de Ticket + initiation de Payment, qui ne doit jamais laisser la base dans un état incohérent en cas de défaillance. Les contraintes d'intégrité référentielle (clés étrangères) garantissent également qu'aucun ticket orphelin ne peut exister sans événement ou utilisateur valide.

**Redis** est utilisé pour quatre usages distincts : (1) stockage des sessions JWT côté serveur (clé `session:{user_id}`, TTL égal à l'expiration du token), (2) clés d'idempotence pour les webhooks Stripe (clé `idempotency:stripe:{payment_intent_id}`, TTL 24h) afin d'éviter le double traitement en cas de livraison dupliquée, (3) cache des listings d'événements publics (clé `events:list:{page}:{filters_hash}`, TTL 60s) pour absorber les pics de trafic, et (4) compteurs de rate limiting par IP et par user (clé `ratelimit:{endpoint}:{ip}`, fenêtre glissante 1 min).

**S3 / MinIO** stocke les objets binaires non structurés : bannières d'événements (`events/banners/{event_id}/{uuid}.{ext}`), exports CSV des participants (`exports/participants/{event_id}/{timestamp}.csv`) et futurs documents attachés. La convention de nommage préfixée par `event_id` permet une politique de cycle de vie S3 ciblée par événement (suppression automatique des assets à l'archivage).

**RabbitMQ** transporte les événements asynchrones inter-services : `ticket.confirmed` (TicketService → NotificationService + StatsService), `payment.failed` (PaymentService → NotificationService + TicketService), `event.cancelled` (EventService → TicketService + NotificationService + RefundService). Ces événements sont détaillés en Phase B.

## Données sensibles RGPD

| Entité | Champ | Catégorie | Mesure de protection |
|---|---|---|---|
| User | email | Donnée directement identifiante | Chiffrement applicatif AES-256 au repos ; pseudonymisation pour analytics |
| User | password_hash | Credential d'authentification | bcrypt cost ≥ 12 ; jamais exposé en API |
| User | first_name, last_name | Donnée directement identifiante | Chiffrement AES-256 ; durée de rétention 3 ans post-dernière connexion |
| User | phone | Donnée directement identifiante | Chiffrement AES-256 ; optionnel, collecte minimisée |
| Notification | content | Donnée indirectement identifiante | Durée de rétention 1 an ; pas d'export analytics |
| RefreshToken | token_hash | Credential de session | Hash SHA-256 + sel applicatif ; purge 30j post-expiration |

> Pour la politique complète de protection des données, durées de rétention et exercice des droits (accès, rectification, effacement), voir **§9 — Conformité RGPD**.
